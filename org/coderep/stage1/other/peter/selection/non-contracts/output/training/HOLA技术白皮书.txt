一、企业内容管理

商业信息以多种形式存在：文本、电子表格、图像、XML 文件、网页、视频、音频、电子
邮件、即时消息和固定内容（比如报告、记录和扫描的图片）。从工程制图、制造流程到中
间市场和销售介绍，非结构化内容对于企业顺畅高效运行是非常重要的。

一个企业内容管理系统提供面向非结构化信息的管理，它依照用户定义的规则提供文档的创
建、管理、处理、分发、归档，系统在内容间建立关联，允许同一个内容被用于不同的上下
文和翻译文档中。增加了智能化，创建分类和元数据，以使搜寻和恢复更为迅速高效。自动
的处理内容的生命周期，通过多渠道促进内容发布。促进部门和信息孤岛式系统的整合。

HOLA是一个健壮的、灵活的平台，通过变化融合以适应企业对内容管理的需要。HOLA平
台易于客户化定制以适应特殊的商业需求或是建立自定义的内容应用。

HOLA系统是提供开发和运行服务的平台，这种服务支持不同应用软件的共同需要。提供丰
富的接口，使开发人员能够把焦点放在业务问题上，使企业能够把应用集成为一个完整的企
业价值链。


1.	无处不在的内容管理

普适内容管理是随时随地管理所有内容类型的能力。HOLA能够使用所有已知格式保存内容
文件，包含丰富的媒体类型，并且可以容易的扩展到新的类型。HOLA能够本地化地涵盖几
乎所有类型的知识--文档，web内容，XML，多媒体，报告，交互内容（包含及时消息，
讨论，电子邮件以及其他）。HOLA系统能够处理内容管理的所有阶段，从创建，管理，转
发，归档到销毁，这些流程已经被法律和公司规章所规定。普适内容管理也意味着与其它组
件的有效协同工作，如操作系统，关系数据库系统，web 应用服务器，验证服务，以及企
业应用，如ERP 和CRM。HOLA无缝连接到这些系统，并且提供了一个分布式的储存库，
以便公司可以访问并且部署他们的内容。




2.	对内容生命周期的管理

HOLA系统从内容的创建、捕捉开始到最终的归档文件，这样一个全程的内容管理。这个目
标可能是向企业网站发布产品信息,可能是向转包商发布工程规范,可能是通过无线设备向领
域代表传递新的销售动机,或者可能是向客户传送货品计价。最终目标可能是归档案或者对
超过生命期的内容的销毁。

HOLA系统的核心性能足以支持每个阶段生命周期，包括内容协作和自动业务工作流。
     
     
     
     
     
     
     
     
     
     
     
     
     
     
     
2.1内容的创建和捕捉

内容管理系统的第一个工作是收集相关内容并添加到内容存储库中，内容有很多来源，可以
是公司的内网和外网。

HOLA与Microsoft Office工具集成让使用者能够直接从内容存储库中添加和重新获取内
容文件。集成使用标准的协议和接口WebDAV 。




2.2与创作工具的集成

HOLA也提供从不同的数据源聚集和导入海量内容的工具，包括ERP/CRM 系统等其他企
业应用。支持文档扫描，把重要的纸制信息转化为电子内容，便于内容管理系统的管理。

HOLA提供一个供用户间沟通、协调、合作的地环境。用户可以使用在线讨论、邮件一起工
作，同时发挥内容管理在内容存储和内容控制特性上的优势。用户可以很容易的做到，把内
容保存在一个HOLA存储库。
     
2.3内容管理

内容存储库是HOLA内容管理系统的基础。内容存储库是一个安全的存储区域，提供有组
织的内容存取，不用考虑内容的格式和来源。HOLA可以使用已知所有格式保存内容。存储
库遵循一套标准属性或可扩展属性以适应每一内容项目。这些属性适合作为描述内容的元数
据。存储库用元数据组织内容，用户可以用它搜索相关的内容。这套存储的属性，每一项都
是可配置的可充分扩展的。

每条在存储库中的项目都被强大而灵活的安全策略所保护，安全策略可以控制什么人可以存
取内容以及每个人所具有的存储级别。HOLA能够做到基于用户或角色的存取控制。

HOLA提供对相同内容的多个版本进行控制，管理和跟踪。提供防止混乱的对文档进行保护
的签入和签出的能力。它可以创建文档的PDF格式用来在各种通道或者设备传输，并且用
来在原始文档被修改时自动更新这些转换。

HOLA可以管理在相关内容和对等内容之间的联系，这种文档称为引用文档。例如，一个
Microsoft Office 文档可能包含对其他Microsoft Office 文档的引用。

HOLA存储库支持对企业范围的数据存储缩放和管理的工具，包括分布式存储，负载平衡，
备份和恢复。

2.4交付内容

HOLA 为用户和内容应用提供访问的方法。用户可以从其它系统访问它们。

HOLA可以通过客户端应用、门户系统和企业应用集成使内容发挥效用。

HOLA可以使一个公司的内容发布到所有渠道中，而使用相同来源的同一个内容。




用户可以对内容进行订阅。当存储库中用户订阅的感兴趣的信息更新了，系统自动通知用户
。

2.5内容归档

今天的公司需要能够以值得信任的、可升级的、节省成本的模式来保护内容。HOLA提供一
个灵活架构可以集成任一归档和存储系统。因为HOLA存储库依赖于底层的操作系统和数
据库，HOLA透明地支持任何面向文件系统接口存储系统。

2.6内容处理

很多企业的内容资产，遵循这样一种内容生命周期之路：内容的创建、回顾、修改、审核然
后被使用，到最终的替代或丢弃。HOLA可以自动设置内容生命的阶段，自动处理每个阶段
的业务进程。工作流规范了在业务流程中的每一个步骤；生命周期则定义了在文档整个周期
中各个阶段发生变化需要遵从的业务规则（如起草、审阅、激活、作废等）。HOLA可以定
义并自动化与创建和分发文档相关的业务流程。例如企业可以自动化文档工作流和生命周期
过程来满足记录管理的规范和ISO 认证过程。



二、HOLA的系统逻辑结构
HOLA系统采用平台模式构建，使其具有良好的组件结构，软件复用程度高，架构灵活易扩
展。下面是HOLA系统的金字塔型逻辑结构：

 

整个系统是以消息通信为基础的，在系统内部通过消息机制来传递信息，任何信息-邮件，
通知，任务，流程等-都是系统的一个消息，这一部分是实现各种用户级应用的基础。同时
，在这一部分实现了一个标准的邮件服务器，提供POP3和SMTP服务，普通的邮件客户
端皆可访问。

基于通信系统，HOLA提供了一系列的协同工具，包括：日程管理、任务管理、公告通知、
联系人管理等，满足日常的工作要求。

文档管理系统构建于协同工具之上，它提供强大而完善的文档管理功能，包括文档的版本管
理、访问日志跟踪、安全设置等，提供各种文档共享的手段，并且能够进行文档流转。

以上这些就提供了基本的功能，能够满足大多数的需求。


对于特定的业务流程，可以在业务流程管理中进行定制，实现个性化的需求。

作为金字塔的塔尖，在下面这些功能很好地利用的基础上，就能够提供出丰富的统计报表和
报告，提供决策支持。
     


三、HOLA系统技术架构
HOLA系统具有良好的层次结构，可分为数据访问层、平台层、业务逻辑层、访问控制层以
及表现层。

 
     
数据访问层（Data Layer）是HOLA系统和数据存储系统之间的桥梁，通过这一层可以将
数据永久化到存储系统中，并且从存储系统中提取数据。可以利用的存储系统包括关系数据
库系统和文件系统。数据访问层为上层应用提供透明的数据存取操作。

平台层(Platform Layer)提供了HOLA系统的基本组件和通用功能，包括各种协同工具，
基础的对象结构，通用的状态模式等。这些组件可以为上层应用提供基本的框架。

业务逻辑层(Business Layer)提供了HOLA系统的所有业务逻辑相关的组件，并且可以进
行扩展。

访问控制层(Access Layer)提供各种协议解析引擎，为客户端和服务器端的交互提供动力
，主要包括基于XML/SOAP方式的访问接口支持，基于WebDav方式的数据访问接口支
持以及DHTML方式的交互。通过该层提供的接口调用，可以实现与第三方软件的集成。
表现层提供所有用来控制和输出到各种用户设备的接口，可以使用访问控制层所提供的各种
引擎，包括浏览器，WebDav客户端等支持。

HOLA 系列产品，包括后端服务、开发接口，以及适应企业对内容和流程管理需求的应用程
序。

1.	内容引擎

内容引擎提供管理企业内容和客户自定义业务对象的软件服务。内容引擎管理范围广泛的企
业内容和对象， 包括电子文档、网络内容、记录、文件夹、工作流定义、登录模板、预定
义权限、分析报告。内容引擎提供的主要服务包括：

	用于对象管理的分布式内容库服务，包括文档、文件夹和订制对象
	分布式的内容库服务，内容存储在不同存储系统内，并可检索
	缓存服务
	面向对象的、可扩展的元数据模型
	版本管理
	关系管理
	服务器端事件和订阅
	内容分类框架和 内容自动分类
	内容转换成 PDF 和 阅读 格式
	文档生命周期模型
	搜索服务，包括基于内容搜索的全文本索引
	授权服务，提供最精细的对象访问控制
	使用业界标准目录产品的身份认证
	审核
	导入和导出
	WebDAV 提供者（和 WebDAV 的客户，如 Microsoft Office，
Macromedia DreamWeaver MX，以及Adobe 的 GoLive 6 配合工
作）
	XML Web Services API






内容引擎包含的服务和模块是：

	对象存储服务（Object Store Service） –管理一个或更多对象存储所
需的数据库事务。一个对象存储 (Object Store) 是储存对象（如文档、
文件夹和业务对象）和定义对象类的元数据的容器。对象库中的每个
Object Store 都关联到一个类 (Class)。Class 是创建对象用的模板或
定义。你通过这个模板或定 义，指定属性(元数据)行为和用到该对象的安
全性。一个对象存储服务，能对 HOLA 域里的多个 Object Store进行存
取，而且一个对象存储服务的所有实例 (Instance) 对于 HOLA 域内的所
有对象 存储器都是可用的。对象存储服务，负责把新的内容加入一个固定
的内容存储器，（通过属性）对对象 进行检索，（通过文件存储服务，或
固定的内容提供者）取回内容，并提供多种内容管理能力。 

	安全性和事件审核（Security and event auditing） –为授权访问对象
和内容提供精细的安全模 式；与第三方的目录服务产品（如微软的 Active 
Directory，LDAP）交互，对用户进行认证，对用户和群组账户数据进行
检索。审核功能使 HOLA系统上活动的监听得以实现。对象发生的任何动
作，都会引发记录为审核条目，供日后分析之用。 

	搜索（Search） –提供基于属性和基于内容的搜索能力。内容引擎搜索能
力使用户能同时搜索多个对象存储器。 

	自动的阶层分类架构（Autoclassification Framework）–根据收到的
某一类型的文档，自动为它指定一个文档类 (Document Class)。提供用
户自定义格式的分类支持，以便创建可订制的分类工具。 

	事件框架（Event Framework） – 当存储在内容引擎服务器上的对象发
生特定活动时，有能力设定 回应的行动。比如，当一个新文档产生时，你
可以配置事件，以启动一个工作流来进一步处理该文件。 

	库服务（Library Services） –库服务，包括文档版本控制、自动内容分
类、文档生命周期管理和元数 据管理。 

	内容相关服务（Content-related Services） –  即文件内容管理部件
的逻辑集合。对象存储服务与这些 部件交互，来进行内容的读、写和全本
文索引。 

	文件存储服务（File Store Service） –管理一个或多个用于存储内容的
文件存储器。对象存储服务 与文件存储服务沟通，把新内容加入固定的内
容存储器。文件存储服务也管理和  HOLA 全本文搜索 引擎的交互。 

	索引服务（Indexing Service） –管理与  HOLA 全本文搜索引擎的交
互。通过文件存储服务，将索 引信息存储到文件系统。 

	内容缓存服务（Content Cache Service） –系统可将内容缓存，以备
以后 的调阅恢复。 

	发布 PDF 插件服务（Publishing PDF Plug-in Service） –处理将文
档转换成 Adobe Acrobat PDF 格 式的请求。将格式转换请求提交给转
换引擎。 

	Java API SOAP 监听程序（Java API SOAP Listener） –接受由内容 
Java API 调用产生的一串需求。监听程序确定与对象进行通讯的 接口和
方式，以及被调用的方式，调整被传递参数的有效性，安排调用引起的 out 
参数返回。 

	WebDAV 提供者 –内容引擎的 IS API 扩展功能，让使用者创建并编辑
文档，通过 WebDAV 兼容应用程 序，如 MS Word 或 DreamWeaver
，管理内容引擎上的文件。 

	CE 和 PE 网络服务（CE and PE Web Services） –实现通过网络访问
内容和流程的功能。

	内容和流程 WebServices API 为开发者提供一套完整的接口，它支持网
络环境下的客户应用开发。这两 个接口有许多相似之处，比如：
◆	两者都是 WS-I(WebServices-互通性) Basic Profile 兼容。
◆	两者使用相同的标准 WS-Security 身份识别机制，允许客户提供
验证（用户名/密码），通过客户企 业目录服务（如 Microsoft 
Windows Active Directory 或 Novell eDirectory），从网上
得到认证。
◆	两者实现了 SOA 架构，并按无状态操作而设计。也就是说，对 
Web Server 的每个请求，都独立于 其他任一请求；请求一旦完成
，服务器将处在与客户特性无关的状态。这个架构的特点是使客户请 
求能在多台主机间平衡加载，实现可扩展性和高效率。 

	内容引擎 web 服务是个底层的 API，它体现了内容引擎服务器的全部功能
。除了基本的文档管理功能 外，所有的制定和管理功能一应具备。它具备
先进的批处理能力，任意复合数据集的检索和更新，均能 在服务器的单一
循环周期中完成。这些能力的综合，是开发功能强大而高度有效的应用程序
的基础。 

	企业管理程序（Enterprise Manager）–通过 它你能管理内容引擎服务
和对象存储器。管理人员使用德雅科技的企业管理（EM）处理下列工作：
◆	创建并管理对象存储器和文件存储器。
◆	管理 HOLA 域名等级服务，例如设置身份识别提供者。
◆	导入/导出对象和定义。
◆	设定 分类规则。 

2.	流程引擎

正和管理由应用、企业用户或外部 用户（如合作伙伴和客户）实现的工作流。流程引擎的
主要服务包括：
	群组工作管理
	事件日志
	身份识别和授权
	用于分析的统计数据收集
	使用 Web Service向第三方系统提供集成服务
	带直观图标的增强型设计工具
	用户自定义模板
	扩展的系统步骤

3.	应用引擎
应用引擎是是内容和流程的展示层。应用引擎为 Workspace和内容引擎间进行的用户身份
识别提供保护，并 提供 SSL 安全措施（在已配置的前提下）。

 

应用引擎包括的服务和应用有：
	Workspace –这是一个终端用户的网络应用环境，为文件管理提供了存取
功能并与 HOLA 的业务流程管理功能紧密结合。 
Workspace在J2EE 应用服务器上的Web Container 内运行。 
Workspace利用Velocity转换很多用户界面元件，例如：查询结果和文
件夹内容。Workspace提供的大部分功 能是零下载的HTML，即能在任
意平台、配合众多的浏览器产品运行。高度图形化的高级功能通过插件 的
形式来提供，包括：在线阅读、在线编辑、在线标注等。 

	对Microsoft®  Office的集成（Integration for Microsoft®  Office
）–使用户在HOLA 内容库内轻松地管理Office文档。用户可直接从
Microsoft  Office目录中存储、搜索和检索文档。 

	内容和流程 Java API（Content and Process Java API） –用于编
制订制应用和扩展的开箱即用应用 程序的 Java 类扩展集合。这些类为内
容和流程引擎的交互提供了可编程的接口。开发人员可以使用这些API 建
立不同的应用程序，包括那些基于 J2EE Web Container (Java 
Servlets)、Enterprise Java Beans (EJB) Container，或是分立的 
J2SE 的 Java 应用程序。 

	Web 应用工具包（Web Application Toolkit）–提供一个可扩展的框架
和可重用的模块，建立网络应用。工具包为让程序开发者访问内容引擎、流
程引擎。为可扩展应用程序的身份识别、事件安排、状态信息、参数选择、
本地化提供行为和数据结构。
     





	应用集成工具包（Application Integration Toolkit） 
应用集成工具包是个全功能 API ，可使第三方把他们基于 Microsoft 
Windows 的客户应用程序与 HOLA 进行集成。HOLA使用这个工具包提
供对 Microsoft Excel、PowerPoint、Word 和 Outlook 的集成。 
HOLA DxCapture 软件也使用这个工具包。 
工具包包含了一套安装在 Windows 客户端的 COM 组件，以及在 J2EE 
应用服务器上运行的 Java servlets。 IntegrationServlet——它支持
来自胖客户端应用程序和基于网络的瘦客户端的请求。这些部件向开发人员
提供了完整的内容管理功能。
	IntegrationServlet服务器 
IntegrationServlet 是个应用引擎服务，许多应用程序用它访问电子文档
、检入 内容、访问工作任务、浏览/搜索对象。IntegrationServlet 支持
来自胖客户端应用程序和基于网络的瘦客户端的请求。

 

1	外部应用程序向 IntegrationServlet 发送命令。胖客户使用基于 XML 的协议
与 IntegrationServlet 进 行沟通，而瘦客户则使用基于请求的命令回应 URL
。
2	IntegrationServlet 处理身份识别，把命令映射为 servlet 请求，并把请求转
给 服务器。
3	服务器用XML数据进行回应。
4	用户提交一个页面。服务器向外部应用程序返回servlet响应。

	配置和订制（configuration and customization） 
可通过修改站点和用户参数进行配置，满足应用程序的需要。 
站点偏好特性适用于所有用户。用户选项是终端用户可在Workspace内设定的配
置选项，适用于个别用户。当升级到新版本或安装新的补丁级别时，使用站点和用
户偏好特性的 Workspace 配置是完全向下兼容的。
◆	改变用户进入 Workspace 时最初看到的内容，比如，用户开始工
作时可以处在搜索或浏览模式，也 可以是任务模式。
◆	改变用户浏览和搜索时看到的内容：可以是文件夹，也可以是内容快
照方式的搜索。
◆	定义不同角色访问者访问的方式。用户可以基于其访问角色在 
Workspace 内访问特 定的内容。一个特定角色的成员也可以明确
地准予或拒绝访问。

四、HOLA的功能特性
1.	集中管理
文档获取 - 文档的集中管理
企业内容管理第一步，就是对所有文档的创作、批阅、审核、发布到归档进行全周期管理，
自动转发、自动归档，对企业的信息知识进行识别、沉淀，将工作过程中形成的文档、资料
、报告、单据、甚至电子邮件，经过科学的评估检测，按照极为规范的方式储存于文档系统
中，因此，任何人都可以快捷方便的查找所需要的信息。

组织文档结构
支持无限层次的文件夹和子文件夹结构。 
支持对档案柜、文件夹、文档进行安全设置，权限级别包括：完全控制、修改、阅读、不可
访问和匿名访问。安全设置可以继承到子文件夹。 





批量文档上传、下载
可以将多个文档一并上传到HOLA系统中，也可以将多个文档一并下载。支持文件夹下载
，可以一次性下载一个文件夹以及子文件夹下的文档，并保持文件夹结构不变。

数据移植 – 整理企业已有资源
如果公司以前有自己的文件服务器，而且已经有了一定的文件量的积累，重新使用HOLA
系统是否意味着以前的文件都没用了呢？答案当然是否定的，文档的积累对于一个企业来说
是一个知识库，使用HOLA文档导入工具帮助客户将历史数据顺利地导入到HOLA系统里
面。可导入的数据可以是原有文件服务器上的内容，也可以是其他信息或文档应用软件中的
内容。

智能文件夹/文档自动归档
系统在每个工作空间创建特定的智能文件夹，将文档上载到此智能文件夹，并指定文档类型
，输入属性值，系统将根据文档的属性值自动将其归档到智能文件夹规则所指定的文件夹下
。智能文件夹规则是以自定义属性为基础动态定义文档归档条件和目标文件夹，通过对自定
义属性进行判断，得出是否复合条件以及是否存在要归档的文件，进而产生相应的归档行为
。同时，智能文件夹可以绑定一个邮件地址，向这个地址发送带有文档附件的邮件，系统自
动将附件文档从邮件中拆分出来，保存到智能文件夹中。




在线编辑
系统需支持在线新建Office文档（Word，Excel，PPT）、在线编辑、电子签名、电子盖
章及修改痕迹保留等。
痕迹保留可让阅读者方便查看什么人什么时间对文档做了什么动作（添加、修改、删除）。

 
网络文件夹
通过从WebDAV客户端应用程序（例如Windows上的“网络文件夹”）进入HOLA，用
户可以管理其工作空间、文件夹及文档。用户可以在Microsoft Word、excel、
PowerPoint中创建/编辑某一文档，并将其直接保存到HOLA中。使用Windows资源管
理器来进行批量上传、下载，方便用户使用。
 

纸质文档电子化
可以将公司的纸质文档（如合同、发票、以及公司原有的重要或需要保存的纸质文档等）转
换为电子文档，便于存储、保存、查找、调用。
支持文档扫描、索引及文档上载到文档系统。支持文档的成批上载及与扫描仪的直接接口。
对于所有被扫描的文档，HOLA支持英文及中文的OCR。
 HOLA使每个文档归属于某一文档类型。用户可以为每个文档设计索引卡片。索引卡片将
被提供给执行文档上载的人员，以便在文档上载时该信息可以被添加。

 
文档同步 – 自动备份
HOLA支持将企业员工个人电脑的文档目录和HOLA文档服务器中的指定目录同步。同步
支持全面的版本管理。具备相应权限的用户也可将服务器中的文档同步到本地的电脑。自动
同步大大减少用户上载文档的步骤，并确保企业共享文档和个人电脑文档的统一，确保企业
的文档完整性。



2.	安全
灵活、强大、易管理的安全特性
HOLA文档管理系统采用了当今最高端的安全技术为企业保护其知识资产。HOLA系统在系
统层、网络层、数据层，及应用层（文档/工作流/流程）提供多级安全控制措施。

支持非专业人员的管理员权限，不再由系统管理员进行集中的权限管理，避免过多的高级权

细粒度的文档访问控制

配权限；可为其设置不同的安全级别，例如：上传、下载、阅读、修改、版本增加、移动、
删除以及匿名访问等等。
可按照公司的组织结构，为用户分组，成组分配权限，简化权限的分配和管理。系统也支持
直接针对个人授权。

防病毒
所有上载到HOLA文档管理系统里的文档都将经过即时更新的病毒库病毒扫描后才会存储
至服务器中。

限制登陆地点
根据IP地址控制方式，用户登陆地点甚至可以被限制在某个办公室内，或是特定的互联网
来源。

限制客户端程序登录数量
为了保证登陆账号的唯一性，Hola系统对客户端登陆系统的数量进行了限制，不允许同一
账号在不同地点同时登陆。使用同一账号最新登陆的机器将会覆盖早期登陆的机器，早期登
陆的客户端将无法继续使用HOLA系统。

集团与分公司与合作伙伴协作区分隔离
支持多工作区模式，对使用者的操作范围、使用功能、数据查询进行完全隔离。
         



防黑客攻击 – 系统层
HOLA系统构建在Linux操作系统之上，采用J2EE架构应用服务器作为系统运行环境。
Linux操作系统是公认的具有较高安全性的操作系统，根据用户应用的需求，可以定制稳定
的核心系统，达到长期运行无故障，并且始终保持较高的运行效率。同时，Linux系统的安
全机制与Unix系统完全相同，文件系统级的用户安全策略可以有效防止病毒和黑客的攻击
，并且极大地降低了被攻击后的损害程度。

Linux的文件系统采用日志型文件系统分区格式，在系统异常崩溃时，可以有效保护数据，
最大程度地减少文件损坏。

J2EE架构的应用服务器是当今业界最为领先的企业级应用服务器模型，经过多年的发展，
J2EE架构已经非常成熟，是世界上主流的企业应用环境。J2EE架构具有自己完整的对象
容器管理机制，安全机制，可以有效避免来自网络上的恶意请求。

 
防黑客攻击 – 网络层
HOLA系统服务采用标准的HTTP协议，仅仅通过HTTP端口即可完成所有的功能。HTTP
端口是互联网上的通用浏览端口，HTTP协议是文本协议，所以传输的信息都是以文本的形
态进行的，这就可以有效避免黑客程序的植入。对那些与HOLA系统无关的网络端口，将
全部被关闭。

防黑客攻击 – 数据传输加密
HOLA系统可以提供基于SSL的信道加密措施，SSL采用128位加密算法，使网络传输的
信息变成密文，这样，即使信息被截获，按照现在的技术也很难被破译。

防黑客攻击 – 先进的用户密码技术
HOLA系统有严谨的用户密码策略，可以对密码做详细的要求，如：要求长度，要求包含各
种字符集（英文、数字、符号等），还可以要求定期更改密码。

      
防灾难性数据丢失 – 服务器备份
HOLA系统的数据和文档可以进行定期定时的自动备份，备份可以是在单一的服务器上的不
同硬盘之间备份，也可以在网络上的不同服务器之间备份。

防错误删除 – 回收站机制
HOLA系统提供回收站机制，任何文档，邮件或其它信息被删除的时候都首先进入回收站
中，直到清空回收站时才真正被删除，有效减少由于用户误操作造成的信息损失。

 
防外泄-控制邮件收发
用户可被限制为只在局域网内部收发邮件，还是可以收发外部的邮件。

 

操作追溯 – 完整文档访问日志
HOLA自动记录完整的审计线索历史，随时记录系统中每个用户进行的所有操作，包括操作
者、操作日期及时间以及操作行为。

      
数据完整性 – 文档版本管理
所有文档阶段性的历史版本都将存档起来，供将来查阅。HOLA支持高级的版本管理功能，
包括版本回滚、文档同步等功能。

 
文档同步 – 自动备份
HOLA支持将企业员工个人电脑的文档目录和HOLA文档服务器中的指定目录同步。同步
支持全面的版本管理。具备相应权限的用户也可将服务器中的文档同步到本地的电脑。自动
同步大大减少用户上载文档的步骤，并确保企业共享文档和个人电脑文档的统一，确保企业
的文档完整性。

3.	应用集成接口
作为文档管理软件，HOLA提供良好的应用集成接口，可以很方便地与其它系统进行衔接。

HOLA提供基于SOAP协议和XML技术的Web Service开发接口。从外部的使用者的角
度而言，Web服务是一种部署在Web上的对象/组件，它具备以下特征：

	完好的封装性，Web服务既然是一种部署在Web上的对象，自然具备对象
的良好封装性，对于使用者而言，他能且仅能看到该对象提供的功能列表。 

	松散耦合，这一特征也是源于对象/组件技术，当一个Web服务的实现发生
变更的时候，调用者是不会感到这一点的，对于调用者来说，只要Web服
务的调用界面不变，Web服务的实现任何变更对他们来说都是透明的，甚
至是当Web服务的实现平台从J2EE迁移到了.NET或者是相反的迁移流
程，用户都可以对此一无所知。对于松散耦合而言，尤其是在Internet环
境下的Web服务而言，需要有一种适合Internet环境的消息交换协议。
而XML/SOAP正是目前最为适合的消息交换协议。 

	使用协约的规范性，这一特征从对象而来，但相比一般对象其界面规范更加
规范化和易于机器理解。首先，作为Web服务，对象界面所提供的功能应
当使用标准的描述语言来描述(比如WSDL)；其次，由标准描述语言描述的
服务界面应当是能够被发现的，因此这一描述文档需要被存储在私有的或公
共的注册库里面。同时，使用标准描述语言描述的使用协约将不仅仅是服务
界面，它将被延伸到Web服务的聚合、跨Web服务的事务、工作流等，
而这些又都需要服务质量(QoS)的保障。其次，我们知道安全机制对于松散
耦合的对象环境的重要性，因此我们需要对诸如授权认证、数据完整性(比
如签名机制)、消息源认证以及事务的不可否认性等运用规范的方法来描述
、传输和交换。最后，在所有层次的处理都应当是可管理的，因此需要对管
理协约运用同样的机制。 

	使用标准协议规范，作为Web服务，其所有公共的协约完全需要使用开放
的标准协议进行描述、传输和交换。这些标准协议具有完全免费的规范，以
便由任意方进行实现。一般而言，绝大多数规范将最终有W3C或OASIS
作为最终版本的发布方和维护方。 

	高度可集成能力。由于Web服务采取简单的、易理解的标准Web协议作
为组件界面描述和协同描述规范，完全屏蔽了不同软件平台的差异，无论是
CORBA、DCOM还是EJB都可以通过这一种标准的协议进行互操作，实
现了在当前环境下最高的可集成性。

Web Service 是一种新的web应用程序分支，他们是自包含、自描述、模块化的应用，可
以发布、定位、通过web调用。 Web Service可以执行从简单的请求到复杂商务处理的
任何功能。一旦部署以后，其他Web Service应用程序可以发现并调用它部署的服务。   

Web Service是一种应用程序，它可以使用标准的互联网协议，像超文本传输协议(HTTP)
和XML，将功能纲领性地体现在互联网和企业内部网上。可将Web服务视作Web上的组
件编程。 
	
通过使用HOLA的Web Service接口，可以很方便地和其它系统集成，无需知道HOLA
的开发语言和开发平台，就可以提取HOLA中的文档，调用HOLA中的功能。其它系统和
HOLA的分界点就是HOLA的接口功能：其它系统欲进入HOLA系统提取信息，通过
HOLA接口进行调用；其它系统欲向HOLA系统提交信息，也通过HOLA接口调用，提交
数据。

　　
　　 

4.	支持客户端的扩展
HOLA系统的公共接口采用XML,Web Service技术封装，为系统提供了广阔的客户端扩
展空间，目前已经实现了通过Web Service接口开发的HOLA Dx插件。
     
5.	分布部署
HOLA系统具有良好的集群技术架构，能够很好地满足这方面的要求。
在客户端和数据库之间是基于J2EE结构的应用服务器层，在此可以建立集群的服务器结构
，支持高并发访问的要求。
由于这种结构，HOLA可以实现基于网络的cluster的应用分布结构。
　5.1基本结构










HOLA的基本结构使用应用服务器、数据库服务器、文件系统和用户终端组成，应用服务器
负责响应用户终端的请求，返回结果；数据库服务器提供应用服务器运行的数据管理服务；
文件系统存储文档。这样的一个结构就可以让HOLA正常运转起来。
根据HOLA的特性，我们可以将系统配置成集群模式，以满足更大规模的应用环境。集群
模式可以分为应用服务器集群和数据库集群两级。

　5.2应用服务器集群

 


应用服务器集群就是将应用服务器安装到多台主机上，同时运行，用户终端可以从任意一台
主机登录，访问HOLA系统。这样的结构可以极大提高系统的并发访问效率，实现负载均
衡。而且，我们可以采用较廉价的主机设备来满足高性能的需求，不必一味提高单点服务器
的性能，有利于系统的随时扩展。这样，就可以有效突破单台服务器的性能瓶颈，有可以很
好地保护原有的投资的重复利用。

　
　5.3分布式文档存储
     
 

分布式文档存储是HOLA的特性之一，就是说HOLA中的文档保存在一个网络存储空间。
任何可以被共享出来的网络空间，都可以在HOLA系统中使用。通过配置，可以将不同用
途的文档（档案柜）存储到相应的网络硬盘上，而不必关心该空间的具体的物理位置，用户
可以完全透明地存取文档。通过这样的结构，可以使HOLA的文档存储空间极大地扩充，
突破单一文件系统的空间限制，基本上可以保证用户的海量文档存储空间。

通过上面的结构分析，我们可以看出，HOLA的并发性是由两级集群技术保障的，首先通过
应用服务器的集群实现大量用户的并发访问，再通过数据库服务器的集群来满足大量应用服
务器的并发请求。同时，HOLA的分布式文档存储特性，有助于提高文档存取的I/O效率
和存储空间扩展，能够很好地均衡大量用户的使用负荷。
